# تحليل مشروع لوحة تحكم OTP (Vercel)

## مكونات المشروع الحالية

| الملف | الوظيفة |
|-------|---------|
| **index.html** | واجهة واحدة: حالة واتساب، رمز QR، أزرار التحكم، معلومات النظام، سجل |
| **config.js** | عنوان API السيرفر فقط: `window.OTP_API_BASE = 'https://76.13.58.90/otp'` |
| **vercel.json** | إعادة توجيه كل المسارات إلى index.html (SPA) |
| **.gitignore** | استثناء node_modules و .vercel |

## ما يطلبه المشروع من السيرفر (Backend)

اللوحة تتصل بهذه المسارات على السيرفر فقط:

- `GET /otp/api/whatsapp/status` — حالة واتساب والرقم وعدد الرسائل
- `GET /otp/api/whatsapp/qr` — رمز QR عند الحالة `qr_pending`
- `POST /otp/api/whatsapp/restart` — إعادة تشغيل واتساب (مع `X-Admin-Key`)
- `POST /otp/api/whatsapp/logout` — تسجيل خروج واتساب (مع `X-Admin-Key`)
- `POST /otp/api/otp/send` — إرسال OTP تجريبي
- **`GET /otp/api/health`** — يُستخدم لعرض **«حالة السيرفر»** ووقت التشغيل وقاعدة البيانات

## لماذا تظهر «حالة السيرفر: متوقف»؟

اللوحة تعتبر السيرفر **يعمل** فقط عندما يكون ردّ `/api/health` كالتالي:

- **HTTP 200**
- **Body بصيغة JSON** ويحتوي على: **`status: 'ok'`**

في أي حالة أخرى تعرض **«متوقف»** أو **«خطأ اتصال»**:

| ما يحدث | ما تراه في اللوحة |
|---------|-------------------|
| طلب `/api/health` فشل (شبكة، CORS، انقطاع) | **🔴 خطأ اتصال** |
| الطلب نجح لكن الـ JSON ليس فيه `status: 'ok'` (مثلاً `status: 'error'` أو لا يوجد `status`) | **🔴 متوقف** |

أسباب محتملة لظهور **«متوقف»**:

1. **تطبيق Node (otp-system) غير شغال على السيرفر**  
   — Nginx يوجّه `/otp/` إلى المنفذ (مثلاً 5050). إذا العملية متوقفة، قد يرجع Nginx 502/503 أو صفحة خطأ، والـ JSON لن يكون بالشكل المتوقع.

2. **مسار `/api/health` يرجع خطأ أو صيغة مختلفة**  
   — مثلاً السيرفر يرجع `{ status: 'error', message: 'database down' }` أو لا يضع `status: 'ok'` عند نجاح الفحص. هنا اللوحة تعرض «متوقف» حتى لو الخدمة تعمل جزئياً.

3. **قاعدة البيانات (Supabase) غير متصلة**  
   — إذا كان كود الـ health يفحص DB ويرجع `status: 'error'` عند فشل الاتصال، ستظهر «متوقف».

4. **CORS**  
   — إذا السيرفر لا يسمح لموقع Vercel، المتصفح يمنع قراءة الرد؛ الطلب يعتبر فاشلاً فترى **«خطأ اتصال»** وليس «متوقف». ظهور «متوقف» يعني عادة أن الطلب وصل والرد كان JSON لكن بدون `status: 'ok'`.

## ما الذي ينقص المشروع؟ (من ناحية الواجهة)

- **تسجيل دخول/جلسة:** لا يوجد؛ أي شخص يعرف الرابط يمكنه فتح اللوحة. حماية الـ API تتم بمفتاح الأدمن (`X-Admin-Key`) فقط.
- **تغيير عنوان API من الواجهة:** لا يوجد؛ التغيير يكون في `config.js` وإعادة نشر على Vercel.
- **عرض رسالة الخطأ من `/api/health`:** تمت إضافته — إن وُجدت `h.message` أو `h.error` تظهر في سجل العمليات عند عرض «متوقف».

## ماذا تفحص على السيرفر لمعرفة سبب «متوقف»؟

1. **التأكد من أن تطبيق OTP يعمل (مثلاً PM2):**
   ```bash
   ssh root@76.13.58.90
   pm2 list
   pm2 logs otp-system
   ```
   إن كان متوقفاً:
   ```bash
   cd /var/www/otp-system && pm2 start server.js --name otp-system
   ```

2. **التحقق من رد `/api/health` من السيرفر نفسه:**
   ```bash
   curl -s http://127.0.0.1:5050/api/health
   ```
   أو عبر Nginx:
   ```bash
   curl -sk https://76.13.58.90/otp/api/health
   ```
   يجب أن ترى JSON مثل:
   ```json
   { "status": "ok", "database": "connected", "uptime": 123, "environment": "production" }
   ```
   إذا رأيت `"status":"error"` أو لا يوجد `status: "ok"` — هذا سبب ظهور «متوقف».

3. **التحقق من Nginx وتوجيه `/otp/`:**
   ```bash
   cat /etc/nginx/sites-available/my-firebase-app | grep -A20 "location /otp"
   sudo nginx -t && sudo systemctl reload nginx
   ```

4. **CORS:** تأكد أن تطبيق Node يرسل رؤوس CORS (مثلاً `Access-Control-Allow-Origin: *` أو نطاق Vercel) لطلبات `/api/*`. بدونها ستظهر «خطأ اتصال» وليس «متوقف».

---

**ملخص:** «حالة السيرفر: متوقف» تعني أن اللوحة حصلت على رد من `/otp/api/health` لكن القيمة `status` في الـ JSON ليست `'ok'`. لمعالجة السبب، تحقق من تشغيل تطبيق OTP على السيرفر ومن أن `/api/health` يرجع `status: 'ok'` عند سلامة الخدمة وقاعدة البيانات.
